Highlight #1 Too many files!

Are you a designer, web developer, writer etc.? If so, you have probably created a print/web ready file that is about to be published/printed when... the dreaded last minute change occurs, it goes off for final approval and there's another change and so on. You file directory might look something like this:

  • graphicdesign_PRINT.pdf
  • graphicdesign_PRINT_Final.pdf
  • graphicdesign_PRINT_final2 then 3 then 4 and so on.

You end up with multiple files that might only have a spelling change but you don't want to get rid of the file just in case you need it because you know when you delete that file the client will want to revert back.

What happens if there are multiple people working on the same files? How do you to keep track of the most current file? This is where we introduce Version Control Systems (VCS); a system that records changes to file(s), who made the change, what they changed, when it changed and compares changes with other users and reverts back to a specific time period if needed.

There are multiple VCS, some work on Unix-like, Microsoft, Mac OS X and Linux etc. Some software is free and others charge a license.

Highlight #2 Collaboration made easy.

A typical project would have a graphic designer, web developer, software manager, writer, project administrator and/or project manager etc. When any one of these professionals works on their own computer system this is known as a Local Version Control System. Can they all work seamless on a project with the same files over many continents and time zones? Yes, with the use of Centralized Version Control System (CVCS) or a Distributed Version Control System (DVCS).

CVCS vs DVCS which is better? A CVCS allows you to synchronize, track and back-up and commit files to a centralized location. You would pull any changes to be made from a central server. Then you make your change and you commit these changes back to the central server for others to see. Whereas, DVCS allows you to collaborate with others and share a snapshot of the changes made. With DVCS you have access to the whole file structure/repository and are able to work locally and offline if needed. You only need to be online to share your changes.

Highlight #3 What is Git?

Git is a free and open source distributed version control system designed for small scale projects to extremely large projects with multiple users. Git is used locally so there is not the need to always be connected to the internet, this streamlines projects. Everyone working on a project has a full history of the repository on their system. Git tracks these files by organizing and storing snapshots of content differences with embedded metadata: what changed, who changed what, when it changed and a description of why it changed. Project files can be edited simultaneously by multiple users and merged into one by pushing the files to a centralized repository.

Each user is able to have independent multiple local branches with an endless number of workflows. When the user is ready to commit the changes they can be 'pushed' to the remote repository. The user doesn't have to push the whole repository just the changes they have made.

Highlight #4 Workflow

Git has three main states that your files can be in: committed, modified and staged.

Committed means the data is stored locally in the Git directory.
Modified means you have changed the file but it isn't staged.
Staged means you are making a modified file ready for your next snapshot.

There are three main sections of a Git workflow: the Working Tree/Directory, the Staging Area, and the Git Directory/Repository.
The Working Tree/Directory is where files are modified.
Staged files occurs when you add snapshots to the staging area.
In the Git Directory/Repository you permanently commit a snapshot.

Files are seen as either tracked or untracked. Tracked files are files taken from the last snapshot whether they are unmodified, modified or staged. Conversly untracked files are files that were not in the last snapshot or staging area.

An example process in Git looks like this:

  • You create a new local Git repository;
  • next, add a new file to the repository (UNTRACKED);
  • when ready to TRACK the file we add it to the STAGING area and take a snapshot/picture;
  • when ready to COMMIT this file for the first time (the snapshot from the STAGING area);
  • we work on it some more and we add new files, both files need to be added to the STAGING area;
  • when ready we COMMIT the changes to the REPOSITORY.

Highlight #5 Using simple Git commands.

Git can be in the command line in Terminal on a MAC, using Command Prompt on Windows or in a graphical user interface.

These are some examples of common Git commands:

  • git init - creating a new Git repository;
  • git log - looking at the repository history, how many commits preformed, date and the author;
  • git checkout --filename - replaces changes in your working tree with the last snapshot;
  • git pull - updating to the newest commit;
  • git merge branchname - merging one branch to the master branch;
  • git add - there are multiple ways to add a list of files, all types of files such as .txt in the current directory or sub directories or even adding a whole project;
  • git commit -m - adding a snapshot to the top of the master branch timeline;
  • git status - tells you what branch you are on.
